# claude-code-thy 提示词工程规划

## 1. 文档目的

本规划文档描述 `claude-code-thy` 的完整提示词工程设计，目标是为项目建立一套：

- 可维护
- 可调试
- 可扩展
- 与 provider 解耦
- 提示词资源与 Python 代码分离

的提示词体系。

这份文档不是“写一段更强的 system prompt”，而是定义：

- 提示词文件放在哪里
- 代码如何加载和组装
- 哪些信息应该进入 prompt
- 哪些信息本来就应该走结构化 API 字段
- Anthropic-compatible 和 OpenAI Responses-compatible 两条线路如何注入
- 后续如何调试与观测

---

## 2. 总体原则

### 2.1 提示词正文不写死在 Python 脚本中

所有主链提示词正文都存放到独立的 `.md` 文件中。

- Python 不保存大段自然语言提示词
- Python 只保存 section 顺序、变量名、装配逻辑、渲染逻辑
- 调 prompt 时优先修改 `.md` 文件，而不是改 provider、runtime、query engine

### 2.2 提示词工程是“构建管线”，不是“大字符串”

系统不应把所有规则堆成一个长字符串，而应拆成明确层次：

- 固定 system sections
- 动态模板 sections
- provider-specific sections
- system context
- user context
- provider renderer

### 2.3 provider 无关

上层统一先产出 provider 无关的 `PromptBundle`，再由各 provider renderer 负责适配：

- Anthropic-compatible 如何注入
- OpenAI Responses-compatible 如何注入

上层不感知具体请求体字段细节。

### 2.4 可调试、可观测

必须可以方便看到：

- 当前启用了哪些 prompt sections
- 每个 section 来自哪个 `.md` 文件
- 动态变量替换结果
- 最终 provider 请求中实际注入了什么

### 2.5 动态模板也使用 `.md`

环境信息、skills 摘要、MCP 指令、用户上下文、项目上下文等动态文本，也应以 `.md` 模板存在。

推荐模式：

- `.md` 文件中保存自然语言模板
- Python 负责收集变量
- renderer 负责变量替换

---

## 3. 当前项目存在的问题

当前 `claude-code-thy` 的主链提示词能力还比较弱，主要表现为：

- 没有独立的 `PromptBuilder`
- 没有清晰的 `system prompt / system context / user context` 分层
- provider 侧主要是在发送消息历史和工具 schema
- 没有统一的 section 概念
- 没有 prompt dump 或 prompt preview 调试出口
- 没有把环境信息、MCP 指令、用户规则、项目规则拆开管理

这会带来几个问题：

- 提示词增强后容易散落到 provider、runtime、query engine 多处
- provider 兼容逻辑容易和提示词逻辑缠在一起
- 调 prompt 时看不到最终注入结果
- 调整一小段提示词就要改 Python 代码

├── context.py
├── loader.py
├── registry.py
├── renderers.py
├── types.py
├── sections/
│   ├── 00_identity.md
│   ├── 10_workflow.md
│   ├── 20_tool_usage.md
│   ├── 25_skill_usage.md
│   ├── 30_permissions_and_safety.md
│   ├── 40_output_style.md
│   └── 50_verification_and_reporting.md
├── templates/
│   ├── 10_environment.md
│   ├── 20_mcp_instructions.md
│   ├── 30_user_context.md
│   └── 40_project_context.md
└── providers/
    ├── anthropic.md
    └── openai_responses.md

同时建议预留项目级覆盖目录：

```text
.claude-code-thy/prompts/
├── overrides/
├── append/

```

用途如下：

- `overrides/`：覆盖内置同名 prompt 文件
- `append/`：附加项目专属规则
- `disabled/`：按文件名或 section id 禁用内置 section

## 5. 各目录职责

### 5.1 `sections/`

放稳定、通用、偏静态的系统级规则。

典型内容：

- 角色定位
- 编码工作方式
- 工具使用原则
- skill 使用原则
- 安全与权限原则
- 输出风格
- 验证与汇报要求

这些文件应尽量少依赖动态变量。

### 5.2 `templates/`

放带变量的动态提示词模板。

典型内容：

- 当前环境信息
- 当前 MCP server 指令
- 当前用户上下文
- 当前项目上下文

这些模板使用占位符，由 Python 渲染。

### 5.3 `providers/`

放 provider 专属补充规则。

例如：

- Anthropic-compatible 的少量注入补充
- OpenAI Responses-compatible 的少量注入补充

要求：

- 这里只放 provider 差异化文本
- 不放主链通用规则
- 不把大段业务提示词转移到 provider 文件中

### 5.4 Python 模块

- `types.py`：定义 `PromptSection`、`PromptTemplate`、`PromptBundle`、`RenderedPrompt`
- `loader.py`：读取 md 文件、解析元数据、缓存文件内容
- `registry.py`：管理默认启用规则、顺序和覆盖关系
- `context.py`：收集环境、skills、MCP、用户上下文、项目上下文变量
- `builder.py`：组装 `PromptBundle`
- `renderers.py`：将 `PromptBundle` 适配为不同 provider 的请求结构
- `bundle.py`：提供统一的外部调用入口

## 6. md 文件规范

建议每个 `.md` 文件支持简单 front matter，避免把 section 元数据写死在代码里。

示例：

```md
---
id: identity
kind: system_section
order: 0
enabled: true
cacheable: true
---

你是 claude-code-thy 的主链编程助手。
你的任务是帮助用户完成代码分析、调试、修改和验证。
```

推荐字段：

- `id`：唯一标识
- `kind`：`system_section` / `template` / `provider_section`
- `order`：拼接顺序
- `enabled`：默认是否启用
- `cacheable`：是否允许缓存

如果希望实现更轻量，也可以先按文件名排序，例如：

- `00_identity.md`
- `10_workflow.md`
- `20_tool_usage.md`

但从长期维护角度看，front matter 更合适。

## 7. system sections 设计

主链 system prompt 建议由以下 7 个核心模块构成。

### 7.1 `00_identity.md`

职责：

- 定义模型在本项目中的身份
- 定义主任务边界
- 明确这是 coding agent，而不是纯闲聊机器人

### 7.2 `10_workflow.md`

职责：

- 收到任务后先理解再执行
- 改代码前先读代码
- 先基于本地事实判断
- 优先选择最直接、最清晰的实现

### 7.3 `20_tool_usage.md`

职责：

- 什么时候优先用 `read / grep / glob / browser / bash`
- 什么时候应避免 `bash` 代替专用工具
- 如何看待并行工具调用

这里强调的是**工具使用策略**，不是重复列出工具 schema。

### 7.4 `25_skill_usage.md`

职责：

- 说明什么情况下应优先考虑使用 `skill` 工具
- 说明 `skill` 依赖运行时自动发现，不应猜测不存在的 skill 名称
- 说明 `skill` 工具的作用是把某个 skill 展开成可执行 prompt，而不是把 skill 当作普通静态文案
- 说明当已有内置工具已经足够直接时，不要滥用 `skill`
- 说明模型应基于当前自动发现到的可用 skills 选择，而不是依赖硬编码列表

这个文件承载的是**整体的 skill 使用规则说明**，不是 skill 列表，也不是每个 skill 的摘要。

### 7.5 `30_permissions_and_safety.md`

职责：

- 权限确认策略
- 文件修改、命令执行、外部调用的边界
- 对 prompt injection、恶意工具结果、危险操作的处理要求

### 7.6 `40_output_style.md`

职责：

- 输出语言和表达风格
- 简洁、直接、可读
- 文件路径、命令、错误信息如何呈现

### 7.7 `50_verification_and_reporting.md`

职责：

- 修改后尽量验证
- 没验证就明确说明
- 最终汇报要说明做了什么、没做什么、风险点是什么

## 8. 动态模板设计

动态模板只承载真正需要自然语言注入的信息，不承载本来就存在于 API 结构化字段中的信息。

### 8.1 `10_environment.md`

变量来源：

- 当前工作目录
- shell
- 操作系统
- 当前日期
- 当前 provider
- 当前模型

### 8.2 `20_mcp_instructions.md`

变量来源：

- 当前 MCP server 连接状态
- MCP server 提供的 instructions

这里的重点是 server 级使用说明，不是重复 MCP 工具的 `name / description / input_schema`。

MCP 工具本身已经会被包装成普通工具 schema 发给模型，所以这部分不再重复写入 prompt。

### 8.3 `30_user_context.md`

变量来源：

- `CLAUDE.md`
- 用户长期偏好
- 工作方式约束

### 8.4 `40_project_context.md`

变量来源：

- 项目级说明
- 工作区补充约束
- 可选的只读仓库快照信息

## 9. 自动发现与模板承载对照

下面这张表用于明确：哪些能力本来就由运行时自动发现并注入，哪些内容才应该交给 `.md` 模板承载。


| 项目                      | 来源                                      | 当前是否自动发现              | 模型主要通过什么途径感知                                             | 是否需要单独 `.md` 模板           |
| ----------------------- | --------------------------------------- | --------------------- | -------------------------------------------------------- | ------------------------- |
| 本地 skills 列表            | `.claude-code-thy/skills` 下的 `SKILL.md` | 是                     | `PromptCommandRegistry` 动态发现 + `skill` 工具的动态 description | 不需要                       |
| MCP skill 列表            | MCP runtime 缓存                          | 是                     | 运行时动态发现 + `skill` 工具的动态 description                      | 不需要                       |
| skill 使用规则              | 固定规则文件                                  | 否                     | 主链 system prompt                                         | 需要，放在 `25_skill_usage.md` |
| 本地工具 schema             | 内置 Tool 定义                              | 是                     | provider 请求体里的 `tools` 字段                                | 不需要                       |
| MCP 工具 schema           | MCP tool 定义包装结果                         | 是                     | provider 请求体里的 `tools` 字段                                | 不需要                       |
| MCP server instructions | MCP 连接对象上的 `instructions`               | 有数据源，但不会自动进入主链 prompt | 需要由 PromptBuilder 主动收集并注入                                | 需要                        |
| 环境信息                    | cwd / shell / os / date / model         | 需要代码运行时采集             | 需要由 PromptBuilder 渲染为文本                                  | 需要                        |
| 用户上下文                   | `CLAUDE.md`、用户偏好                        | 需要代码运行时采集             | 需要由 PromptBuilder 渲染为文本                                  | 需要                        |
| 项目上下文                   | 项目规则、工作区补充约束                            | 需要代码运行时采集             | 需要由 PromptBuilder 渲染为文本                                  | 需要                        |
| 工具使用原则                  | 设计规则，不是运行时数据                            | 否                     | 主链 system prompt                                         | 需要                        |
| 安全、权限、输出风格规则            | 设计规则，不是运行时数据                            | 否                     | 主链 system prompt                                         | 需要                        |


这张表的核心结论是：

- **skill 列表不靠 md 模板维护**
- **skill 使用规则可以通过单独 md 文件注入**
- **工具 schema 不靠 md 模板维护**
- **只有规则类文本和上下文类文本，才交给 md 模板**

## 10. 什么信息该进 prompt，什么不该进 prompt

这是实现时必须严格遵守的边界。

### 10.1 应进入 prompt 的内容

- 角色和行为规则
- 输出风格
- 安全与权限规则
- 工具使用策略
- MCP server instructions
- 用户上下文
- 项目上下文
- 环境信息

### 10.2 不应重复进入 prompt 的内容

- 本地工具的完整 `name / description / input_schema`
- MCP 工具的完整 `name / description / input_schema`
- 已经通过 provider `tools` 字段发送的结构化 schema

原因：

- 结构化工具定义已经由 API 提供给模型
- 再转成自然语言复述会造成 prompt 膨胀
- 容易造成 schema 与文本描述不一致

关于 skills，还需要单独说明一层：

- 本地 skills 的发现应由 `PromptCommandRegistry` 自动完成
- 当前 session 可用 skill 名称和简述，应由运行时动态读取
- 模型感知这些 skills 的主要通道，应是 `skill` 工具自身的动态 description，而不是额外再放一个 skills 摘要模板
- 同时可以提供一个独立的 `skill` 规则文件，专门告诉模型“何时、如何、在什么边界内使用 skill”

也就是说：

- `claude-code-thy/.claude-code-thy/skills` 中的内容由代码自动扫描
- skill 名称、描述、可调用性来自运行时动态注入

结论：

- **工具 schema 走结构化字段**
- **工具使用原则走 prompt**
- **MCP server instructions 走 prompt**
- **skills 列表走运行时自动发现与动态注入，不单独做摘要模板**
- **skill 使用规则走独立的 system section**

## 11. 中间数据结构设计

建议在 Python 中定义 provider 无关的中间结构。

示意：

```python
PromptBundle(
    system_sections=[...],
    template_sections=[...],
    provider_sections=[...],
    system_context={...},
    user_context={...},
    debug_meta={...},
)
```

建议至少定义：

- `PromptSection`
- `PromptTemplate`
- `PromptBundle`
- `RenderedPrompt`

其中 `RenderedPrompt` 应代表 provider 最终消费前的结果，例如：

- `system_text`
- `user_context_text`
- `provider_text`
- `messages_patch`

## 12. provider 注入策略

### 12.1 Anthropic-compatible

建议注入方式：

- 将 system sections 渲染后放入原生 `system` 字段
- 将 user context 作为一条 meta user message 插入消息头
- 保留原始 `messages`
- 保留原始 `tools`

### 12.2 OpenAI Responses-compatible

建议注入方式：

- 将 system sections 渲染后放入 `instructions` 或等价系统字段
- 将 user context 转成最前面的 meta input item
- 正常传 `input`
- 正常传 `tools`

### 12.3 renderer 职责边界

renderer 只负责：

- provider 请求结构适配
- system / user context 的注入位置映射
- provider 专属补充文本的注入

renderer 不负责：

- 写提示词正文
- 决定 section 内容
- 决定业务规则

---

## 13. 变量渲染策略

不建议引入 Jinja2 这类复杂模板引擎。

原因：

- 增加依赖
- 调试复杂
- 容易写出过度动态模板

建议使用受控的简单占位符替换，例如：

- `{{ workspace_root }}`
- `{{ current_date }}`
- `{{ model_name }}`
- `{{ user_context }}`
- `{{ mcp_instructions }}`

建议 renderer 只支持：

- 字符串替换
- 缺失变量保护
- 基本的列表合并与空值裁剪

不支持：

- 任意表达式执行
- 模板内复杂控制流

## 14. 完整实现内容

这套提示词工程应作为一个完整能力整体实现，包含以下内容。

### 14.1 资源层

- 建立 `src/claude_code_thy/prompts/`
- 建立 `sections/`、`templates/`、`providers/`
- 建立项目级 `.claude-code-thy/prompts/` 覆盖目录

### 14.2 装载层

- 实现 `.md` 文件读取
- 实现 front matter 解析
- 实现文件缓存
- 实现 section 排序和启用逻辑

### 14.3 上下文层

- 收集环境变量
- 收集 provider/model 信息
- 收集 MCP instructions
- 收集用户上下文和项目上下文
- 调用运行时已有的 skill 发现能力，而不是在 md 模板中重复维护 skills 列表

### 14.4 组装层

- 组装 `PromptBundle`
- 合并静态 sections、动态 templates、provider sections

- Anthropic-compatible renderer

- 输出最终 system/instructions/messages 注入结果

### 14.6 调试与观测层

- CLI `prompt dump`
- Web `prompt preview`
- section 级别查看
- token 估算
- 渲染结果摘要

## 15. 调试与可观测性

### 14.1 CLI

建议新增命令：

```bash
claude-code-thy prompt dump --session <session_id>
claude-code-thy prompt dump --session <session_id> --provider anthropic-compatible
claude-code-thy prompt dump --session <session_id> --provider openai-responses-compatible
```

输出内容应包含：

- section 列表
- section 来源文件
- 渲染后的正文
- 最终拼接结果
- provider 注入方式

### 14.2 Web API

建议新增接口：

- `GET /api/runtime/prompt-preview?session_id=...`
- `GET /api/runtime/prompt-sections?session_id=...`

返回内容应包含：

- 当前 `PromptBundle`
- 当前启用 sections
- 当前动态变量
- provider 渲染结果摘要

### 14.3 统计项

建议同时输出：

- section 字符数
- section token 估算
- 最终 system prompt token 估算
- 缓存命中情况

## 16. 明确不建议的做法

### 15.1 不要把大段提示词继续写在 provider 中

provider 应只做：

- 请求体结构适配
- 历史消息映射
- tools 序列化
- system/user context 注入

不应保存大量业务提示词正文。

### 15.2 不要让 `QueryEngine` 直接拼自然语言

`QueryEngine` 负责流程，不负责提示词写作。

### 15.3 不要把完整工具 schema 再写成自然语言摘要

工具的 `name / description / input_schema` 已经有结构化传输通道。
不要在 prompt 中重复复述。

### 15.4 不要把所有动态信息都塞进 system prompt

要区分：

- 哪些是 system sections
- 哪些是 dynamic templates
- 哪些是 system context
- 哪些是 user context
- 哪些只应该留在结构化 API 字段中

### 15.5 不要引入过重的模板引擎

优先使用简单、可控、易 debug 的占位符渲染。

## 18. 架构总结

这套提示词工程的目标结构如下：

```text
Markdown Prompt Files
        ↓
Prompt Loader
        ↓
Prompt Registry
        ↓
Prompt Context Builder
        ↓
Prompt Bundle Builder
        ↓
Provider Renderer
        ↓
Anthropic / OpenAI Responses Request
```

这个结构建立后，提示词资源、上下文收集、provider 注入、调试观测会彼此解耦。后续无论是增强 prompt 内容、增加项目级覆盖、优化上下文注入，还是扩展调试能力，都不需要再把逻辑散落到 provider、runtime、query engine 中。
