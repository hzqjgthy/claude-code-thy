# claude-code-thy 会话双轨日志系统开发规划

## 1. 文档目的

本文档用于重新规划 `claude-code-thy` 的会话日志系统。

本次规划不再以“只保留结构化 JSONL”作为主目标，而是明确采用你已经选定的 `C` 方案：

- 一份给人直接阅读的 `.log`
- 一份给程序精确分析的 `.jsonl`

两份日志同时生成、同时追加、绑定同一个 session。

本次规划的最终目标是让每个会话都能产出类似下面这样的日志文件对：

```text
/Users/thy/Desktop/thy_skills/Claude_Code/claude-code-thy/.claude-code-thy/session-logs/20260428_164512__452106da-d065-465f-a3d9-85c53b324845.log
/Users/thy/Desktop/thy_skills/Claude_Code/claude-code-thy/.claude-code-thy/session-logs/20260428_164512__452106da-d065-465f-a3d9-85c53b324845.jsonl
```

其中：

- `.log` 是主入口，强调人工易读、排障效率、错误提示清晰
- `.jsonl` 是底层事实流，强调结构化、可检索、可二次分析

---

## 2. 本次方案的明确结论

这次不采用“单一格式日志”。

最终方案固定为“双轨日志”：

### 2.1 人工日志

- 格式：纯文本 `.log`
- 用途：开发者直接打开阅读、`tail -f`、快速定位错误
- 特点：
  - 每轮会话按大块展示
  - 工具调用和工具结果按顺序展示
  - 错误信息用明显分隔块强调
  - 尽量不要求开发者懂内部 JSON 结构

### 2.2 结构化日志

- 格式：`JSONL`
- 用途：程序分析、后续 Web 调试页、精准检索、回放
- 特点：
  - 每行一条事件
  - 严格时间顺序
  - 保存完整结构化字段
  - 能精确还原运行轨迹

### 2.3 两者关系

- `.log` 和 `.jsonl` 绑定同一个 session
- 文件名前缀完全一致
- 两者都按 session 生命周期持续追加
- `.log` 是给人看的投影
- `.jsonl` 是给程序看的事件流

## 4. 输出路径与命名规则

## 4.1 目标输出目录

这次方案明确以你当前希望的目录风格为准。

推荐默认输出目录：

```text
<workspace_root>/.claude-code-thy/session-logs
```

对当前项目来说，展开后就是：

```text
/Users/thy/Desktop/thy_skills/Claude_Code/claude-code-thy/.claude-code-thy/session-logs
```

原因：

- 与当前项目现有运行产物目录结构一致
- 不把调试日志混进源码目录和 docs/tests
- 与 `.claude-code-thy/sessions`、`.claude-code-thy/tasks`、`.claude-code-thy/tool-results` 风格统一
- 同一个项目的 session、任务、日志都收敛在隐藏运行目录里，后续清理和迁移更简单

## 4.2 文件名前缀规则

每个 session 第一次开始记录日志时，生成一个固定前缀：

```text
<session_start_local_timestamp>__<session_id>
```

例如：

```text
20260428_164512__452106da-d065-465f-a3d9-85c53b324845
```

这里的时间戳建议使用：

- 本地时区时间
- 格式：`YYYYMMDD_HHMMSS`

## 4.3 最终文件名

```text
<prefix>.log
<prefix>.jsonl
```

例如：

```text
20260428_164512__452106da-d065-465f-a3d9-85c53b324845.log
20260428_164512__452106da-d065-465f-a3d9-85c53b324845.jsonl
```

## 4.4 `--resume` 行为

当使用 `--resume` 恢复同一个 session 时：

- 必须继续写原来的 `.log`
- 必须继续写原来的 `.jsonl`
- 不能因为新开进程而重新生成新的时间戳文件

因此，首次生成的日志前缀需要持久化到 session 里。

建议写入：

- `session.runtime_state["log_bundle_prefix"]`
- `session.runtime_state["log_started_at"]`

这样重启后仍能找到同一组日志文件。

## 5. 目录与模块设计

建议新增模块：

```text
src/claude_code_thy/session_logs/
├── __init__.py
├── manager.py
├── paths.py
├── records.py
├── jsonl_writer.py
├── text_writer.py
├── formatters.py
└── serializers.py
```

模块职责建议如下：

- `paths.py`
  负责：
  - 计算输出目录
  - 生成文件名前缀
  - 定位 `.log` / `.jsonl` 路径

- `records.py`
  定义日志事件类型、record 基础结构、事件名常量

- `serializers.py`
  把 `ChatMessage`、`ToolEvent`、`ToolResult`、异常对象、provider preview 转成结构化字典

- `jsonl_writer.py`
  负责结构化日志的逐行追加写入

- `formatters.py`
  负责把统一事件渲染成可读文本块

- `text_writer.py`
  负责 `.log` 的追加写入

- `manager.py`
  对外提供统一的 `SessionLogManager`
  它同时驱动：
  - JSONL writer
  - text writer

不建议把 `.log` 的文本拼接逻辑直接散落在 `runtime.py` 或 `query_engine.py` 里。

## 6. 总体架构

推荐采用“一套事件，两套输出”的设计。

```text
ConversationRuntime / QueryEngine / ToolRuntime / Provider 边界
            ↓
      SessionLogManager.record(...)
            ↓
    ┌───────────────────────────────┐
    │ 统一事件对象 SessionLogRecord │
    └───────────────────────────────┘
            ↓                 ↓
      JsonlWriter         TextWriter
            ↓                 ↓
         .jsonl              .log
```

这样做的核心优势是：

- 采集逻辑只写一遍
- `.jsonl` 和 `.log` 不会内容不一致
- 后期如果要增加第三种输出，比如 HTML 调试报告，也能复用同一事件流

## 7. `.jsonl` 设计

## 7.1 作用定位

`.jsonl` 是底层事实流。

它负责：

- 严格记录事件顺序
- 提供机器可解析结构
- 让后续问题定位不依赖正则去拆文本日志

## 7.2 基础 record 结构

每条记录建议统一为：

```json
{
  "version": 1,
  "session_id": "452106da-d065-465f-a3d9-85c53b324845",
  "turn_index": 3,
  "event_index": 27,
  "timestamp": "2026-04-28T08:12:34.123456+00:00",
  "event": "tool_call_finished",
  "data": {}
}
```

## 7.3 推荐事件类型

建议保留以下核心事件：

- `session_started`
- `session_resumed`
- `turn_started`
- `command_parsed`
- `provider_request`
- `provider_response`
- `provider_error`
- `message_added`
- `tool_event`
- `tool_call_started`
- `tool_call_finished`
- `permission_requested`
- `permission_resolved`
- `turn_finished`
- `runtime_error`

这部分作为底层事实源，字段可以比 `.log` 更全。

## 8. `.log` 设计

## 8.1 作用定位

`.log` 是开发者主读文件。

它必须满足：

- 直接打开就能看懂
- 不需要先理解内部数据结构
- 错误时能快速看到上下文
- 工具调用和工具结果一眼对应
- assistant 输出按原文展示，不做摘要化改写

## 8.2 展示原则

`.log` 的核心不是“尽可能短”，而是“尽可能清楚”。

因此建议：

- 按 session 开头输出基础头信息
- 按 turn 分大块
- 每轮内部按真实发生顺序写
- assistant 文本始终按原文输出
- 工具入参完整保留
- 工具输出在不过长时完整保留；过长时保留头一段和尾一段，中间用省略号连接
- 错误块高亮展示

## 8.3 推荐整体布局

## 9. `.log` 的错误展示规则

这部分是本次方案的重点。

因为你明确要求：

- 要人工易读
- 要有清晰的报错提示

因此，错误不应只是普通一行文本，而应作为单独的高可见块输出。

## 9.1 错误块样式

推荐使用固定样式：

```text
!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
错误
时间: 2026-04-28 16:46:02 +08:00
阶段: provider.parse_tool_arguments
错误类型: ProviderError
错误信息: 工具参数不是有效 JSON object
Provider: anthropic-compatible
相关工具: bash
tool_use_id: toolu_02

相关输入:
{
  "command": "cat > player/snake_game_advanced.html << 'EOF' ..."
}

最近一次请求摘要:
- model: glm-4.5
- endpoint: https://open.bigmodel.cn/api/anthropic/v1/messages
- tools_count: 12

建议排查:
- 检查工具参数是否是完整 JSON object
- 检查长字符串中的引号、反斜杠、换行转义
- 检查流式工具参数是否发生截断
!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
```

## 9.2 错误块必须包含的字段

建议错误块至少包含：

- 时间
- 第几轮
- 阶段
- 错误类型
- 错误信息
- provider
- 工具名
- `tool_use_id`
- 相关输入
- 最近一次请求摘要

## 9.3 错误块与 assistant 错误消息的关系

当前系统在 provider 异常时，会追加一条 assistant 消息，例如：

- `API 调用失败：...`

日志系统不应依赖这条最终消息去推断错误。

正确做法是：

- 先单独写一条结构化错误事件
- 再照常记录最终 assistant 错误消息

这样：

- `.log` 中能看到完整错误块
- transcript 也仍保持现有行为

## 10. 统一事件与文本映射规则

推荐采用以下映射方式：

### 10.1 只在 `.jsonl` 写事件原型

每个事件首先形成统一 `record`。

### 10.2 `.log` 不再自己理解业务对象

`.log` writer 只消费统一事件，不直接消费：

- `ChatMessage`
- `ToolResult`
- `ProviderResponse`

也就是说，业务对象先进入 `serializers.py`，再变成统一事件。

这样有两个好处：

- 业务层和日志展示层解耦
- `.log` 和 `.jsonl` 的口径一致

---

## 11. 事件模型与 `.log` 的对应关系

建议这样对应：

- `session_started`
  写 `.log` 文件头

- `session_resumed`
  在 `.log` 中追加一段“会话恢复”说明

- `turn_started`
  开一个新的“第 N 轮”分隔块

- `command_parsed`
  如果是 slash 命令，在 `.log` 中显示命令名、参数和最终是否转成 `submit_prompt`

- `provider_request`
  在 `.log` 中显示摘要
  不建议直接把完整 request preview 全量灌进主正文
  但建议在需要时写“详细请求预览见 `.jsonl`”

- `provider_response`
  在 `.log` 中显示 assistant 原文和工具调用清单

- `message_added`
  用来写最终落入 session 的消息

- `tool_event`
  以简洁小条目形式写入 `.log`

- `tool_call_started`
  在 `.log` 中单独开“工具调用 N”块，并展示完整输入 JSON

- `tool_call_finished`
  在 `.log` 中展示工具结果

- `permission_requested`
  在 `.log` 中明显写出“等待权限确认”

- `permission_resolved`
  在 `.log` 中写“用户已同意/已拒绝”

- `provider_error` / `runtime_error`
  输出高亮错误块

- `turn_finished`
  输出本轮结束摘要

---

## 12. 建议保留的 `.jsonl` 详细字段

`.jsonl` 里建议完整保留：

- 用户输入全文
- assistant 输出全文
- `content_blocks`
- `metadata`
- 工具调用完整入参
- 工具结果完整 `output`
- 工具结果 `preview`
- 工具结果 `structured_data`
- `ToolEvent.metadata`
- provider 的脱敏 `request_preview`

`.jsonl` 的目标就是“不要丢信息”。

如果 `.log` 因为工具输出过长而做了“头尾保留、中间省略”，`.jsonl` 必须保留完整细节。

- 工具输出过长时的头尾保留内容
- 失败时的错误块
- 结束状态

如果某轮没有工具调用，也要明确写：

```text
[工具调用]
本轮无工具调用
```

这样读日志时不会误以为是漏记。

## 16. 长内容处理策略

这次方案不能把核心信息一刀切地简单截断。

因此建议：

### 16.1 `.log`

默认规则：

- 用户输入
- assistant 文本
- 工具入参

其中：

- assistant 文本始终按原文完整输出，不做摘要，不做中间裁剪
- 工具输出在长度不超过阈值时完整输出
- 工具输出超过阈值时，只保留头一段和尾一段，中间插入明显省略标记

### 16.2 极长块处理

如果遇到特别长的内容，例如：

- 大段 HTML
- 超长 provider request body
- browser snapshot

建议策略不是“删掉”，而是：

- assistant 文本仍然完整展示
- 工具输出按照“头部 + 省略号 + 尾部”方式展示
- 同时补一行：
  - `完整工具输出见同名 .jsonl`

推荐展示形式：

```text
[工具结果 3]
- 状态: 成功
- 摘要: 读取日志文件
- 输出:
<头部内容>

... ... ...（中间已省略，共 128734 字符）... ... ...

<尾部内容>
```

推荐裁剪逻辑：

- 当工具输出长度 `<= tool_output_inline_max_chars` 时，原样输出
- 当工具输出长度 `> tool_output_inline_max_chars` 时：
  - 保留前 `tool_output_head_chars`
  - 保留后 `tool_output_tail_chars`
  - 中间插入省略说明

### 16.3 `.jsonl`

`.jsonl` 应完整保留原始字段值：

- assistant 输出全文保留
- 工具输出全文保留
- 不做 UI 式 200 字符截断
- 不做头尾裁剪

## 17. 并发与一致性要求

## 17.1 同一 session 必须串行写

同一 session 可能同时收到：

- 流式文本增量
- 工具事件
- 消息追加
- 错误事件

因此建议 `SessionLogManager` 以 session 为粒度持有锁，确保：

- `.log` 顺序正确
- `.jsonl` 顺序正确
- 两份文件对同一事件的记录次序一致

## 17.2 先构造 record，再双写

同一事件建议按这个顺序处理：

1. 先生成统一 `record`
2. 先写 `.jsonl`
3. 再把 `record` 格式化后写 `.log`

原因：

- `.jsonl` 作为底层事实流更稳定
- `.log` 只是投影

## 17.3 失败隔离

建议：

- `.jsonl` 写失败时，日志管理器返回错误并至少尝试写 `.log`
- `.log` 写失败时，不影响主流程，也不影响 `.jsonl`
- 日志系统自身错误不能中断主对话流程

更具体地说：

- 日志失败最多追加一条内部告警
- 不能因为日志写入失败让聊天功能挂掉

## 22. 为什么不把 `.log` 作为唯一事实源

虽然 `.log` 是这次方案的主入口，但仍不建议把它作为唯一事实源。

原因：

- `.log` 面向阅读，格式以后可能继续优化
- `.jsonl` 更适合精确检索
- 如果未来要做：
  - Web 调试页
  - 按 `tool_use_id` 查询
  - 自动分析错误聚类
  - 对比两次 provider request

结构化事件流会更稳

所以这次方案的正确姿势是：

- 人平时主要看 `.log`
- 需要深挖时看 `.jsonl`
